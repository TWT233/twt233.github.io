VSCode 结构

VSCode 结构解体得很彻底，非常高度的模块化，从外往内的结构大致是：

  • VS code
    • 应用+UI 框架：Electron
      • 文本编辑器：Monaco Editor
      • UI 管理组件
      • 插件/拓展管理模块(Extension Host)
  • Language Server

可以看到 VSCode 本身结构非常简单，所以大部分的功能其实都是由 Extension 来实现的。而 Extension Host 则负责所有插件的管理，套这一层壳的好处也很明显，一个字就是快，当然这个快是指用户体验上的快：

  • 可以保证插件不影响整体的启动速度（做 Lazy Load 就可以了）
  • 可以保证插件不影响 UI 的响应速度
  • 可以保证插件不直接修改整体 UI

接下来是 Language Server（以下简称 LS），这个可能看起来比较突兀，因为 LS 是独立于 VSCode 的组件，LS 与 VSCode 间的通讯是使用 Language Server Protocol（以下简称 LSP） 进行的。

Language Server/Language Server Protocol

这套系统我以前也没有接触过，查了一会儿相关资料才发现 LS 和 LSP 是 MS 在这几年内推行的一套系统，如字面意所示：LS 是做特定语言代码的支持的；LSP 是 LS 与编辑器间通信的协议；想要实现对某种语言的支持，只要提供对应的 LS 即可。

LS+LSP 可以做到代码解析与编辑器的分离，就算换一个编辑器/IDE，我还是可以用上同一套代码解析工具（只要这个编辑器/IDE 使用的是 LSP）。

而我这个课程设计项目，要做的就是 LS 的拓展开发，我们需要实现一个自己的 LS，进行代码分析，并定位/纠正一些语法和语义错误。
